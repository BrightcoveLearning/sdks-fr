    Migration vers ExoPlayer 2 Framework

    Dans cette rubrique, vous apprendrez comment utiliser le framework ExoPlayer 2 avec le SDK Brightcove Native Player pour Android.

    Présentation

    Le SDK natif Brightcove pour Android utilisant ExoPlayer 1 a été obsolète.

    Si vous le souhaitez, vous pouvez consulter la documentation de référence ExoPlayer 1, mais il est recommandé de migrer vers ExoPlayer 2 Framework.

    Google ExoPlayer est un lecteur multimédia open source, au niveau de l'application pour Android. Brightcove tire parti de l'ExoPlayer pour profiter des avantages qu'il offre. Pour plus d'informations sur les avantages de la mise à niveau vers ExoPlayer, consultez l'article de blog Brightcove Native SDK with ExoPlayer 2 .

    Découvrez comment migrer pour les types de projets suivants :

    • Nouveau projet
    • Projet ExoPlayer 1 existant
    • Projet MediaPlayer existant

    Reportez-vous aux exemples d'applications ExoPlayer pour plus d'informations sur l'intégration du framework ExoPlayer 2 dans votre projet.

    Nouveau projet

    Procédez comme suit pour ajouter ExoPlayer 2 à un nouveau projet Android :

    1. Dans votre fichier build.gradle , ajoutez la dépendance suivante : 
      dependencies {
  implementation "com.brightcove.player:exoplayer2:${anpVersion}"
}
    2. En définissant l' transitive option true pour indiquer à gradle d'inclure automatiquement d'autres dépendances telles que le plugin de lecture hors ligne. Si vous utilisez une version gradle inférieure à 5, les dépendances transitives sont désactivées par défaut. Donc, vous devrez ajouter l'instruction suivante : 
      dependencies {
  implementation "com.brightcove.player:exoplayer2:${anpVersion}" {transitive = true}
}
    3. Si vous utilisez une variable d'environnement pour la version SDK, ouvrez le fichier gradle.properties et définissez la valeur de version actuelle pour le SDK natif pour Android. Il devrait ressembler à ceci : 
      anpVersion=6.5.0

    Projet ExoPlayer 1 existant

    Procédez comme suit pour mettre à jour un projet ExoPlayer 1 existant afin d'utiliser le framework ExoPlayer 2 :

    1. Dans votre fichier constants.gradle , mettez à jour la version Android Build Tools : 
      buildToolsVersion "28.0.2"
    2. Dans votre fichier build.gradle , mettez à jour les bibliothèques de support Android : 
      implementation 'com.android.support:appcompat-v7:${APPCOMPAT_V7_VERSION}'
implementation 'com.android.support:support-v4:${SUPPORT_V4_VERSION}'
implementation 'com.android.support:recyclerview-v7:${SUPPORT_V4_VERSION}'
implementation 'com.android.support:support-vector-drawable:${SUPPORT_V4_VERSION}'

      Les valeurs actuelles des constantes se trouvent dans le fichier constants.gradle .

    3. Dans votre fichier build.gradle , ajoutez le référentiel Google Maven : 
      maven {
  url 'https://maven.google.com'
}
    4. Si votre projet utilise de la publicité avec Google IMA, mettez à jour les annonces Google Play Services : 
      implementation 'com.google.android.gms:play-services-ads:'+PLAY_SERVICES_VERSION
      Il est important de noter ici que le SDK Brightcove Native Player pour Android ne prend pas en charge les applications qui tentent d'utiliser ExoPlayer et ExoPlayer 2.
    5. Dans votre fichier build.gradle , où vous avez précédemment importé la bibliothèque ExoPlayer Brightcove en tant que dépendance comme ceci : 
      dependencies {
  implementation "com.brightcove.player:exoplayer:${anpVersion}"
}

      Changez le nom de la dépendance :

      dependencies {
  implementation "com.brightcove.player:exoplayer2:${anpVersion}"
}

      La valeur actuelle de Brightcove SDK version (anpVersion) se trouve dans le fichier gradle.properties.

    6. En définissant l' transitive option true pour indiquer à gradle d'inclure automatiquement d'autres dépendances telles que le plugin de lecture hors ligne. Si vous utilisez une version gradle inférieure à 5, les dépendances transitives sont désactivées par défaut. Donc, vous devrez ajouter ce qui suit : 
      dependencies {
	implementation "com.brightcove.player:exoplayer2:${anpVersion}" {transitive = true}
}
    7. Si vous utilisez une variable d'environnement pour la version SDK, ouvrez le fichier gradle.properties et définissez la valeur de version actuelle pour le SDK natif pour Android. Il devrait ressembler à ceci : 
      anpVersion=6.5.0

      Aucun codage supplémentaire ne doit être requis pour intégrer ExoPlayer 2 dans votre projet.

    Projet MediaPlayer existant

    Procédez comme suit pour mettre à jour un projet MediaPlayer existant afin d'utiliser le framework ExoPlayer 2 :

    1. Dans votre fichier build.gradle , ajoutez la dépendance suivante : 
      dependencies {
  implementation "com.brightcove.player:exoplayer2:${anpVersion}"
}
    2. En définissant l' transitive option true pour indiquer à gradle d'inclure automatiquement d'autres dépendances telles que le plugin de lecture hors ligne. Si vous utilisez une version gradle inférieure à 5, les dépendances transitives sont désactivées par défaut. Donc, vous devrez ajouter ce qui suit : 
      dependencies {
  implementation "com.brightcove.player:exoplayer2:${anpVersion}" {transitive = true}
}
    3. Si vous utilisez une variable d'environnement pour la version SDK, ouvrez le fichier gradle.properties et définissez la valeur de version actuelle pour le SDK natif pour Android. Il devrait ressembler à ceci : 
      anpVersion=6.5.0
    4. Dans votre application, utilisez BrightCoveExoPlayerVideoView au lieu de BrightCoveVideoView , qui utilise Android Media Player.

    Changement de rupture

    Les modifications suivantes apportées au SDK natif Brightcove pour Android sont des changements incurables avec ExoPlayer 2. L'utilisation de l'ExoPlayer 1 a été obsolète avec le SDK natif Brightcove.

    Écouteurs internes ExoPlayerVideoDisplayComponent

    La ExoPlayerVideoDisplayComponent classe vous permet de définir des écouteurs pour obtenir l'état des composants ExoPlayer. Voici une liste de certains des auditeurs :

    Définir le processus d'écoute Recevoir des informations sur l'état
    SetDebuglistener InfoListener
    SetCaptionListener CaptionListener
    SetMetadatalistener Metadatalistener
    setInternalErrorListener InternalErrorListener

    Voici des recommandations pour l'utilisation de l'ExoPlayer 2.

    InfoListener

    Pour l' InfoListener interface, les modifications suivantes ont été apportées :

    • La onAvailableRangeChanged méthode a été supprimée
    • La onLoadStarted méthode a un length paramètre supplémentaire

    Exemple : ExoPlayerVideoDisplayComponent utilisant ExoPlayer 2

    public interface InfoListener {
void onVideoFormatEnabled(Format format, int trigger, long mediaTimeMs);
void onAudioFormatEnabled(Format format, int trigger, long mediaTimeMs);
void onDroppedFrames(int count, long elapsed);
void onBandwidthSample(int elapsedMs, long bytes, long bitrateEstimate);
void onLoadStarted(int sourceId, int type, int trigger, Format format,
				long mediaStartTimeMs, long mediaEndTimeMs);
void onLoadCompleted(int sourceId, long bytesLoaded, int type, int trigger, Format format,
				long mediaStartTimeMs, long mediaEndTimeMs, long elapsedRealtimeMs, long loadDurationMs);
void onDecoderInitialized(String decoderName, long initializedTimestampMs,
				long initializationDurationMs);
}

    CaptionListener

    L' CaptionListener interface est inchangée.

    Exemple : ExoPlayerVideoDisplayComponent utilisant ExoPlayer v2

    public interface CaptionListener {
void onCues(List<Cue> cues);
}

    Metadatalistener

    L' MetadataListener interface a été introduite avec ExoPlayer v2. L'utilisation précédente de l' Id3MetadataListener est maintenant obsolète.

    Exemple : ExoPlayerVideoDisplayComponent utilisant ExoPlayer v2

    public interface MetadataListener {
void onMetadata(Metadata metadata);
}

    Avec le nouveau, MetadataListener vous recevrez un Metadadata objet dans le onMetadata rappel. Avec l'obsolète Id3MetadataListener, vous avez reçu une liste d'objets Id3Frame. Avec le nouvel écouteur, vous pouvez toujours obtenir une liste d' Id3Frame objets comme indiqué ci-dessous :

    exoPlayerVideoDisplayComponent.setMetadataListener(new ExoPlayerVideoDisplayComponent.MetadataListener() {
@Override
public void onMetadata(Metadata metadata) {
 for(int i = 0; i < metadata.length(); i++) {
		 Metadata.Entry entry = metadata.get(i);
		 if (entry instanceof Id3Frame) {
				 Id3Frame id3Frame = (Id3Frame) entry;
		 }
 }
}
});

    InternalErrorListener

    L' InternalErrorListener interface a de nombreux changements. Veuillez vérifier le code ci-dessous pour la comparaison :

    ExoPlayerVideoDisplayComponent utilisant ExoPlayer v2

    public interface InternalErrorListener {
void onPlayerError(Exception e);
void onAudioTrackUnderrun(int bufferSize, long bufferSizeMs, long elapsedSinceLastFeedMs);
void onLoadError(int sourceId, IOException e);
void onDrmSessionManagerError(Exception e);
}

    ExoPlayerVideoDisplayComponent utilisant ExoPlayer v1 (obsolète)

    public interface InternalErrorListener {
void onRendererInitializationError(Exception e);
void onAudioTrackInitializationError(AudioTrack.InitializationException e);
void onAudioTrackWriteError(AudioTrack.WriteException e);
void onAudioTrackUnderrun(int bufferSize, long bufferSizeMs, long elapsedSinceLastFeedMs);
void onDecoderInitializationError(MediaCodecTrackRenderer.DecoderInitializationException e);
void onCryptoError(MediaCodec.CryptoException e);
void onLoadError(int sourceId, IOException e);
void onDrmSessionManagerError(Exception e);
}

    Lorsque vous utilisez ExoPlayer 2, le type d'exception que vous obtenez onPlayerError(Exception e) est com.google.android.exoplayer2.ExoPlaybackException , mais cela peut changer à l'avenir.

    En utilisant com.google.Android.Exoplayer2.ExoplayBackException, vous pouvez obtenir les types d'exception suivants :

    • ExoPlaybackException.TYPE_SOURCE
    • ExoPlaybackException.TYPE_RENDERER
    • ExoPlaybackException.TYPE_UNEXPECTED

    Vous pouvez également récupérer la cause de l'erreur en obtenant l' Throwable objet avec ExoPlaybackException.getSourceException().

    Prise en charge de la lecture hors ligne

    Si votre projet utilise la MediaDownloadable.setConfigurationBundle(Bundle) méthode pour spécifier les préférences de téléchargement telles que le débit vidéo, veuillez mettre à jour votre code comme suit :

    Ancien format (obsolète)

    com.google.android.exoplayer.MediaFormat

    Nouveau format

    com.brightcove.player.model.MediaFormat

    Classes d'API média héritées

    Les classes d'API média héritées obsolètes, dans le com.brightcove.player.media package, ont été supprimées du SDK natif Brightcove pour Android.

    Si vous utilisiez les classes Media, vous devez migrer et commencer à utiliser la com.brightcove.player.edge.Catalog classe.

    Les cours suivants ont été déplacés de com.brightcove.player.media vers com.brightcove.player.model:

    • DeliveryType.java
    • CaptionType.java
    • ErrorFields.java
    • VideoFields.java
    Si vous utilisez l'une de ces quatre classes, il vous suffit de remplacer le package d'importation et vous serez tous définis.
